These instructions assume that you have successfully run both the export and dist commands on your own codelab, as outlined in my other Codelabs and are ready to commit the artifacts into your preferred serving vehicle, in this case GitHub.

It's also assumed that you have a GitHub account already set up and can log in!

Important: in all cases throughout this tutorial, please substitute <username> (including the angle brackets) with your actual GitHub username

Check for Git package

Run the following command in your terminal to check if the git package is installed:

$ which git

If you get output showing the full path to the git executable, then skip to the next section.

However, if you get no output whatsoever, then you need to install the git package from your distro's repositories.

To do that, run the following command in your terminal:

$ sudo apt update && sudo apt install git

You should now be able to move on to the next section ...

The big green button below will actually take you to the correct GitHub page for creating a new repository: https://github.com/new

Tip: right-click the button and select Open link in new window so that this and that page can be positioned side-by-side

Start by going to Create a new repository

Under Repository name enter <username>.github.io

Once your Repository name is verified as available, leave all other fields with their defaults, then scroll down and click the Create repository button at bottom-right

As soon as GitHub creates your new repository, you'll be presented with a few suggestions to get you started.

If you would like to save these steps for future reference, then copy/paste them into an empty text-file now, before moving on with next steps, because once you navigate away from that page, you'll never see them again!

Obviously, your GitHub account is password protected, which is a good thing. Not only does this protect unauthorized access in a web browser, but also at the command-line.

In order to publish files to your new GitHub repository, you need a secure mechanism to allow only you to access your account, however just typing in your password is not secure enough!

This is where GitHub access tokens come into play. By creating an access token on the GitHub website itself, then using that token in lieu of your password at the command-line, you increase your security-level exponentially.

You may have seen this mentioned in Linux before: With great power comes great responsibility

Tip: once you get to the Personal access tokens (classic) step below, create a Bookmark in your browser, so that you can quickly get back to that page in future ... the bookmark would point to https://github.com/settings/tokens

To create a GitHub access token, you need to go to Developer Settings. To do this, follow these steps:

  • Once logged in, click your profile pic in the top-right corner of the GitHub website
  • Select ✶ Settings about 2/3 of the way down the menu
  • Scroll down and select ⟨⟩ Developer Settings on the left
  • Once in Developer Settings, click the drop-down arrow next to Personal access tokens and select Tokens (classic)
  • Find the Generate new token menu, either at top-right or in the middle of the page, then click it and select Generate new token (classic)

At this point, you may be asked to re-enter your GitHub password ... remember, this is a good thing =)

Once you've reached the New personal access token (classic) section, you are presented with alot of choices, but don't worry ... you're only interested in a few!

  • Start by entering a reference/note regarding this particular access token ... usually something like the current stage of your project. Just for today, type in First commit
  • Next, you want to set an Expiration date ... it's best to keep this as short as possible, so click the drop-down menu and select Custom...
  • Click the little calendar icon for Show date picker, and select tomorrow's date which is the minimum allowed (it will be highlighted with a gray square)
  • Now, at the top of the list (under Select scopes) click the first checkbox to select-all in the repo scope (Full control of private repositories)
  • Scroll down a little, and under the admin:org scope, select only the read:org checkbox (Read org and team membership, read org projects)
  • Scroll down some more, and click the checkbox to select gist (Create gists)

That's it ... now, scroll to the bottom of the list and click the Generate token button

Next, you will be presented with the original Personal access tokens (classic) page, but now it will list your current token(s)

Important: be aware that by coming back to the Personal access tokens page at a later time, the access tokens themselves will not be visible ... so, either keep that browser tab open while you work, or copy/paste the new access token to a temporary file

Please keep this information safe, and do not share access tokens, as that could pose a huge security risk (no so much from your friend/colleague, but from the information transfer)

Remember when GitHub created your new repository, and you were presented with a few suggestions to get you started?

In this section, we're going to follow the suggestions that were outlined under Create a new repository on the command line, but with a few minor changes ...

Open your terminal in (or cd into) the parent directory containing your dist folder, which should be site ... if you don't see a dist folder, you may need to re-run the claat export ... command.

Remove the existing README.md file by running the following command:

$ rm README.md

Now run the following command to create a new README.md file (copy/paste everything, from cat to EOF inclusive, to your terminal):

cat << EOF > README.md
This repository contains the source files for
<BR>
the Codelabs at [https://<username>.github.io](https://<username>.github.io)
EOF

The reason you create your README.md file in the parent folder is that the build and dist folders are removed then recreated by the CLaaT tool multiple times during development. By storing the file outside the dist folder, there is little risk of it also being removed

Note: you will only have to create this README.md file once.

Alternatively, you can just copy/paste this version straight into a new README.md file, using your favorite text editor:

This repository contains the source files for
<BR>
the Codelabs at [https://<username>.github.io](https://<username>.github.io)

Remember to replace <username> with your actual GitHub username

Now, you need to initialize your local dist folder as a Git repository. Initializing as a git repo means preparing the unversioned project so that it's recognized as a legitimate Git repository. The git init command achieves this by creating a (normally hidden) .git folder containing the git "plumbing" and version information

Don't be too concerned about why you need to do this, but rather that it has to be done each & every time gulp dist is executed. As previously mentioned, the build and dist folders are removed then recreated by the CLaaT tool multiple times during development

Open your terminal in (or cd into) your dist folder, and run the following commands:

git init
git add .
git commit -m "First commit"
git branch -M master
git remote add origin https://github.com/<username>/<username>.github.io.git

Note the slight variation from the suggested GitHub commands. In our case, we replaced
git add README.md with git add . (dot) ... the dot means all modified files & sub-folders.
Also, each commit should have a unique message that describes the latest changes. In this case, our commit message matches our access token's Note field, but only to keep it simple

Also, remember that these five commands (or a variation thereof) will need to be run each & every time gulp dist is executed, so now might be a good time to create a bash script containing the commands. Tip: replace "First commit" above with $1 and ensure that your script has been called with this message parameter!

There is a sample script available for download, within the repo for these codelabs ... to view the script, just click the sample script link above, or to save the script to your computer, right-click the sample script link and select Save link as ...

Keep your terminal open, as you will need it for the next step

The final step in publishing your codelab(s) is to push (or upload) your files to the GitHub repository that you created earlier.

This step assumes that you have run the git add | git commit and other commands on your local repo.

To push any changes to your remote repository, ensure that you are in the dist directory within your terminal, and run the following command:

$ git push -f origin master

The -f switch is required, to forcefully overwrite the remote files, since our entire local repo is rewritten each & every time

Once you've pushed your files to GitHub, it's time to check to see if everything works as expected

Tip: once you get to the Pages step below, create a Bookmark in your browser, so that you can quickly get back to that page in future ... the bookmark would point to https://github.com/<username>/<username>.github.io/settings/pages

Start by going to your https://github.com/<username>/<username>.github.io repository on GitHub.

Now click ✶ Settings at top-right (if your window is too small, this may be hidden within an ellipsis menu (…) with the tooltip Additional navigation options). Note that this is not the settings for your GitHub account, but the settings for this repo.

Once in Settings, click Pages under the Code, planning, and automation section at left.

Under the GitHub Pages heading, you should see a message that says Your site is live at https://<username>.github.io/ with a Visit site button to the right. Click the button to visit your newly published website, which should open in a new tab.

If the GitHub servers are busy, it might take a minute or two for your site to come online. If this is your second or subsequent push to GitHub, you'll see a message under GitHub Pages saying Last deployed by <username> x [minutes | hours] ago.
Refresh this page every 15 seconds or so, until the time-frame changes to now. Then, you can view your site using the View site button

You should now be viewing your codelab(s) ... if not, please review the next section for Troubleshooting

Some users have experienced 404 messages when attempting to host a static website on GitHub Pages.

Here are some suggestions that are proven to be successful, depending on your GitHub setup:

Plan A

Open your terminal in (or cd into) the dist folder, then create an empty .nojekyll file by running the following command:

$ touch .nojekyll

Next, commit your changes, give it a minute or two, then check your Pages website to see of this solution helped

Plan B

If the solution above doesn't work, you may need to change a few Settings, specifically for your Pages repo.

Start by going to your https://github.com/<username>/<username>.github.io repository on GitHub.

Now click ✶ Settings at top-right (if your window is too small, this may be hidden within an ... ellipsis menu)

Note that these Settings are specific to this repo, i.e. they are not global

Once in Settings, click the ⊝ Actions drop-down at left, and select General.

Under the Actions permissions heading, select Disable actions then click the Save button just below it.

Give it a minute or two, then check your Pages website to see of this solution helped. You may have to recommit your repo (this suggestion is untested)
